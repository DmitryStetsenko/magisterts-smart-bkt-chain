# Управління проєктом та інтеграція з Jira «Smart-BKT-Chain»

Цей документ описує методологію управління розробкою, структуру епіків у Jira, правила кодування завдань, регламент роботи з Git та критерії готовності функціоналу (Definition of Done).

---

## 1. Система кодування та компонентів (Jira Components)

Для забезпечення чіткої фільтрації завдань, автоматизації аналітики та зручності розробки в межах монорепозиторію, кожне завдання прив'язується до відповідного архітектурного компонента за допомогою префіксів у назві та вибору поля **Component** у Jira.

| Код (Префікс) | Компонент Jira | Сфера відповідальності (Scope) | Робоча директорія |
| :--- | :--- | :--- | :--- |
| **`INF`** | `DevOps-Infra` | Конфігурація монорепозиторію, Docker, CI/CD, хмарний деплой | `/`, `/.github/`, `/docker/` |
| **`BK`** | `Backend-API` | Ядро NestJS бекенду, авторизація, робота з базою даних | `/apps/api/` |
| **`BKT`** | `BKT-Engine-AI` | Математичний рушій BKT, оцінювання коду (LLM Judge), черги | `/apps/api/src/bkt/` |
| **`W3`** | `Web3-Blockchain` | Смарт-контракти Solidity, Hardhat/Foundry, релейєр транзакцій | `/packages/contracts/` |
| **`CF`** | `Client-Frontend` | Клієнтська частина Next.js (кабінет студента, викладача, граф знань) | `/apps/web/` |
| **`AF`** | `Admin-Frontend` | Панель адміністрування Next.js (CMS для контенту, CRM аналітика) | `/apps/admin/` |
| **`SH`** | `Shared-Packages` | Спільні пакети монорепозиторію (Prisma schema, E2E типи) | `/packages/` |

## 2. Реєстр епіків (Jira Epics Registry)

Весь обсяг робіт розділено на **8 ключових епіків**, кожен з яких представляє собою логічно завершений етап або великий блок функціоналу системи.

### `SBC-EP-INF` | [INF] Monorepo & CI/CD Setup
*   **Опис:** Створення інфраструктурного каркаса монорепозиторію, локального середовища розробки та автоматичних перевірок коду.
*   **Ключові результати (Deliverables):**
    *   Ініціалізовано Turborepo з підтримкою npm Workspaces.
    *   Створено спільні конфігураційні файли (TSConfig, ESLint, Prettier).
    *   Розгорнуто Docker-compose конфіг (PostgreSQL + pgAdmin).
    *   Налаштовано GitHub Actions (Linter, Type Check, автоматичні тести).
    *   Запущено Next.js сайт документації `apps/docs` з деплоєм на GitHub Pages.

### `SBC-EP-SH` | [SH] Database Modeling & Shared Packages
*   **Опис:** Створення єдиної схеми даних та типів для гарантування E2E Type Safety.
*   **Ключові результати (Deliverables):**
    *   Описано Prisma схему у `packages/database` (користувачі, курси, навички, BKT-стани, логи відповідей, таблиці сирої телеметрії `TelemetryLog` та поведінкових профілів `BehavioralProfile`).
    *   Виконано перші міграції PostgreSQL.
    *   Створено спільний пакет `packages/types` з типами REST API та телеметрії.
    *   Налаштовано автоматичну генерацію ERD-схеми бази даних.

### `SBC-EP-BK` | [BK] Authentication & Authorization Services
*   **Опис:** Реалізація безпеки платформи, реєстрації та розмежування ролей (RBAC).
*   **Ключові результати (Deliverables):**
    *   Інтегровано SIWE (Sign-In with Ethereum) для верифікації криптогаманців студентів.
    *   Налаштовано JWT / OAuth2 (Google SSO) для викладачів та адміністраторів.
    *   Написано `RolesGuard` у NestJS для захисту ендпоінтів на основі ролей (Admin, Educator, Student, Employer).

### `SBC-EP-BKT` | [BKT] Bayesian Knowledge Tracing Core Engine
*   **Опис:** Розробка математичного алгоритму оцінювання знань студента в реальному часі з адаптивними параметрами BKT та аналізом поведінки.
*   **Ключові результати (Deliverables):**
    *   Реалізовано сервіс розрахунку матриць ймовірностей BKT (із динамічним коригуванням $P(S)$ та $P(G)$ на основі поведінкових профілів).
    *   Розроблено сервіс аналізу телеметрії для обчислення коефіцієнта довіри (Trust Coefficient) та виявлення використання ШІ.
    *   Створено логіку «Холодного старту» (Cold Start) з вхідними діагностичними тестами.
    *   Реалізовано API для обходу та валідації графів знань.

### `SBC-EP-LLM` | [BKT] Automated Code Assessment & LLM Judge
*   **Опис:** Реалізація системи автоматичної перевірки практичних завдань з програмування.
*   **Ключові результати (Deliverables):**
    *   Створено ізольований сервіс виконання unit-тестів коду студента.
    *   Інтегровано Gemini API для семантичного аналізу коду та надання розширеного текстового фідбеку.
    *   Налаштовано черги BullMQ (Redis) для асинхронної обробки перевірок без затримки API.

### `SBC-EP-W3` | [W3] Web3 Infrastructure & Smart Contracts
*   **Опис:** Розробка та деплой смарт-контрактів для верифікації навичок на рівні блокчейну.
*   **Ключові результати (Deliverables):**
    *   Написано та скомпільовано смарт-контракти для dNFT (ERC-721) та SBT (ERC-5192).
    *   Створено систему відновлення токенів у разі компрометації гаманця (SBT Recovery).
    *   Налаштовано релейєр (Relayer) для оплати газу за студентів (gasless transactions).
    *   Створено юніт-тести контрактів у Foundry з покриттям >95% та Fuzzing-тестуванням.

### `SBC-EP-CF` | [CF] Client Web Application Portal
*   **Опис:** Інтерфейс клієнтської частини для студентів, викладачів та працедавців.
*   **Ключові результати (Deliverables):**
    *   Реалізовано дашборд студента з інтерактивним графом знань (Apache ECharts / D3).
    *   Інтегровано модуль збору телеметрії в редактор коду Monaco (темп друку, паузи, copy-paste).
    *   Створено кабінет викладача з візуалізацією успішності та поведінкових профілів (рівнів довіри).
    *   Інтегровано криптогаманець через RainbowKit/Wagmi.
    *   Реалізовано інтерфейс проходження завдань та редактор коду (CodeMirror/Monaco).

### `SBC-EP-AF` | [AF] Administrative CMS & CRM Panel
*   **Опис:** Створення робочого місця адміністратора для управління навчальним контентом та аудиту.
*   **Ключові результати (Deliverables):**
    *   Розроблено візуальний конструктор графа навичок (Skill Graph Builder).
    *   Створено CRM інтерфейс перевірки успішності студентів та університетських доменів.
    *   Реалізовано адміністративну панель виклику SBT Recovery.

## 3. Типи завдань у Jira (Issue Types)

Для ведення розробки використовуються стандартні типи завдань Agile, розширені для потреб R&D проєкту:

*   **`Story` (Користувацька історія):** Опис нової бізнес-фічі з точки зору кінцевого користувача. Має містити Acceptance Criteria (AC).
*   **`Task` (Технічне завдання):** Чисто інженерна робота (налаштування конфігів, баз даних, написання тестів).
*   **`Bug` (Помилка):** Дефект у вже існуючому та протестованому коді.
*   **`Spike` (Дослідження):** Дослідницьке завдання для вивчення нових технологій, алгоритмів або підходів. Завжди завершується написанням текстового документа (ADR) та фіксованим Time-box (наприклад, 4 або 8 годин).

## 4. Правила роботи з Git (Git Flow)

### 4.1. Найменування гілок (Branch Naming)
Кожна гілка повинна починатись із префікса напрямку, номера та короткого опису англійською мовою (через дефіс):
*   `feature/CF-12-knowledge-graph`
*   `feature/W3-45-sbt-recovery`
*   `bugfix/BK-99-jwt-expiration`
*   `spike/BKT-5-cold-start-analysis`

### 4.2. Правила оформлення коммітів (Conventional Commits)
Усі повідомлення коммітів повинні відповідати стандарту Conventional Commits з обов'язковим вказуванням архітектурного префікса (Scope) та номера завдання з Jira:

```text
<type>(<scope>): <опис англійською мовою> <номер_таски>
```

**Дозволені типи (`type`):** `feat` (нова фіча), `fix` (виправлення багу), `docs` (документація), `style` (форматування), `refactor` (рефакторинг), `test` (тести), `chore` (налаштування білду тощо).

**Приклади:**
*   `feat(CF): add interactive ECharts skill graph to dashboard SBC-12`
*   `fix(BK): resolve JWT validation crash on expired token SBC-99`
*   `test(W3): add foundry fuzz tests for SBT recovery logic SBC-45`

## 5. Критерії готовності (Definition of Done - DoD)

Завдання не може бути переведене в статус **Done (Завершено)**, якщо воно не відповідає загальним та специфічним критеріям готовності.

### Загальні критерії:
1.  Код пройшов перевірку лінтера (`npm run lint`) та збірку типів (`npm run typecheck`).
2.  Написано необхідні юніт- та інтеграційні тести (покриття коду новими тестами >80%).
3.  Оновлено відповідні файли документації в директорії `docs/` (якщо функціонал міняє API або архітектуру).
4.  Створено Pull Request, який успішно пройшов автоматичні перевірки в CI/CD (GitHub Actions).

### Специфічні критерії готовності:

*   **Для Frontend (CF, AF):**
    *   Відсутність помилок Hydration Mismatch (елементи Web3 загорнуті в `dynamic` з `ssr: false`).
    *   Інтерфейс повністю адаптивний (перевірено на мобільних роздільних здатностях).
    *   Усі текстові ресурси винесені у файли локалізації `next-intl` (UA/EN).
*   **Для Backend (BK, BKT):**
    *   Для всіх нових API-ендпоінтів додано декоратори Swagger (`@ApiOperation`, `@ApiResponse`).
    *   Усі DTO валідуються за допомогою `class-validator`.
    *   Для операцій з БД використано транзакційні блоки та оброблено помилки унікальних ключів.
*   **Для Web3 & Смарт-контрактів (W3):**
    *   Контракти скомпільовані за допомогою Solidity 0.8.20+ без попереджень (warnings).
    *   100% покриття логіки контракту тестами у Foundry.
    *   Виконано аналіз витрат газу (Gas Report) та оптимізовано змінні сховища (storage slots).
    *   Логіку перевірено за допомогою фазінг-тестів (Fuzzing).
